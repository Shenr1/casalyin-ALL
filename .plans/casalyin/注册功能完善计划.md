# 注册功能完善计划

本文档基于当前注册功能代码现状，列出尚未接入或需完善的功能点，并给出分步骤实现方案。每步可独立完成并验证后再进行下一步。

---

## 〇、图形验证码 & 邮箱验证码：需要配什么？

要让「图形验证码」和「邮箱验证码」真正生效，只需保证下面两项可用，**不需要单独为验证码做配置**。

### 1. 图形验证码

- **依赖**：后端用 **Redis** 存验证码（UUID → 验证码文本），不依赖第三方服务。
- **你需要做的**：
  - 确保 **Redis 已启动**，且与当前环境配置一致。
  - 配置在 **sa-base** 的 `sa-base.yaml` 里：`spring.data.redis`（host/port/password 等），例如 dev 里是 `127.0.0.1:6379`。
- **如何自检**：打开注册页，第一步能出现验证码图片、点击可刷新，说明图形验证码和 Redis 都正常。若一直“加载中”或报错，先看后端日志和 Redis 是否连通。

### 2. 邮箱验证码（发邮件）

- **依赖**：后端用 **Spring Mail** 发邮件，读的是 `spring.mail` 配置。
- **你需要做的**：
  - 在 **sa-base** 的 `sa-base.yaml`（按环境选 `dev`/`test`/`prod` 等）里配置 **发件邮箱（SMTP）**：
    - `spring.mail.host`：如 `smtp.163.com`、`smtp.qq.com`
    - `spring.mail.port`：如 `465`（SSL）
    - `spring.mail.username`：发件邮箱账号
    - `spring.mail.password`：**邮箱的 SMTP 授权码**（163/QQ 等不是登录密码，需在邮箱设置里开 SMTP 并生成授权码）
  - 若用 163：网易邮箱 → 设置 → POP3/SMTP/IMAP → 开启 SMTP 服务 → 生成授权码，把授权码填到 `password`。 

- **如何自检**：在注册页输入邮箱、通过图形验证码后点「获取验证码」，能收到邮件即表示配置正确；若提示“发送验证码失败”，看后端日志里的邮件异常（如认证失败、连接超时）。

- **如何验证「连不上 Gmail 是不是本机/网络导致」**：  
  发邮件时是**运行后端的机器**去连 SMTP（如 smtp.gmail.com），和用户浏览器、用户网络无关。  
  - **方法一（推荐）**：在开发/测试环境调用接口 **`GET /api/auth/test-mail-connection`**。接口会用当前配置的 `spring.mail.host:port` 尝试 TCP 连接（约 6 秒超时）。  
    - 返回成功：说明本机到该邮件服务器**能连通**，问题多半是认证或 SSL 等。  
    - 返回失败（如超时）：说明**本机到该邮件服务器不通**，是运行后端的这台机器的网络/防火墙/地区限制导致，可换 SendGrid 或部署到海外后再用 Gmail。  
  - **方法二**：在本机 PowerShell 执行  
    `Test-NetConnection -ComputerName smtp.gmail.com -Port 587`  
    若 `TcpTestSucceeded : False` 或长时间无结果，同样说明本机到 Gmail 不通。

**小结**：图形验证码 = Redis 可用即可；邮箱验证码 = 配好 `spring.mail` 且用授权码。无需再为“验证码”单独加配置项。

### 3. 图形验证码会不会让用户觉得复杂？可以关掉吗？

- **作用**：图形验证码和邮箱验证码**目的不同**。
  - **邮箱验证码**：证明「这个人能收到这个邮箱」，用来确认身份。
  - **图形验证码**：在点「获取验证码」之前拦住**机器/脚本**，防止有人用程序批量给无数邮箱发邮件（骚扰用户、打爆你的邮件服务）。所以它是「防刷接口」，不是「再验一次人」。

- **会不会复杂**：多一步输入图片里的字符，确实多了一点操作。如果更在意**流程简单**，可以关掉图形验证码，只靠**发送验证码的限流**（同一邮箱/同一 IP 每分钟只能请求一次）来防刷，很多场景也够用。

- **可选方案**：项目里已支持**通过配置开关**「发验证码前是否要图形验证码」：
  - **后端**（casalyin-admin 的 `application.yaml` 或环境变量）：`register.send-email-captcha-enabled`。**默认已设为 `false`**，即不校验图形验证码，注册流程更简单。
  - **前端**（构建环境变量）：`VITE_REGISTER_SEND_EMAIL_CAPTCHA_ENABLED=true` 时才展示图形验证码并传给后端；不设或非 `true` 时不展示。
  - 需要更强防刷时：后端改为 `true`，前端构建时加 `VITE_REGISTER_SEND_EMAIL_CAPTCHA_ENABLED=true`，即恢复「先过图形验证码才能发邮件」。

## 二、分步骤完善清单

### 步骤 1：发送验证码前接入图形验证码（防刷）

**目标**：用户点击「获取验证码」前必须先通过图形验证码，避免接口被脚本刷邮件。

**后端（casalyin-iam）**

1. **发送验证码接口增加图形验证码校验**
   - 在 `SendVerifyEmailForm` 中增加字段：`captchaUuid`、`captchaCode`（或复用 sa-base 的 `CaptchaForm`）。
   - 在 `LoginController` 的 `sendVerifyEmail` 中，先调用 `captchaService.checkCaptcha(...)`，通过后再调用 `emailVerifyService.sendVerifyCode`。
   - 参考：`CaptchaService.checkCaptcha(CaptchaForm)` 会校验 Redis 中的验证码并删除，一次一用。

2. **可选：发送验证码限流**
   - 按邮箱限流：同一邮箱 1 分钟内只能请求一次。
   - 按 IP 限流：同一 IP 每分钟最多 N 次（如 5 次）。
   - Redis key 可沿用 `RedisKeyConst.Support` 或新增，如 `email:send:rate:{email}`、`email:send:rate:ip:{ip}`。

**前端（casalyin-server）**

1. 在注册页第一步（邮箱验证步骤）：
   - 调用 `apiGetCaptcha()` 获取图形验证码图片与 `captchaUuid`。
   - 展示验证码图片 + 输入框，用户输入 `captchaCode`。
   - 点击「获取验证码」时，将 `captchaUuid`、`captchaCode` 随 `email` 一起传给发送验证码接口；若后端返回验证码错误，则刷新图形验证码再让用户重试。

**验收**：未输入或输错图形验证码时无法发送邮件；正确输入后才能收到邮件。

---

### 步骤 2：注册提交前校验图形验证码（可选，与步骤 1 二选一或同时）

**目标**：若希望注册接口本身也防脚本，可在注册时再校验一次图形验证码（当前已有 Turnstile 时，可仅做步骤 1，本步为可选增强）。

**后端**

1. 在 `RegisterForm` 中增加可选字段：`captchaUuid`、`captchaCode`。
2. 在 `UserService.register` 中，若配置了「注册需要图形验证码」，则先调用 `captchaService.checkCaptcha(...)`，通过后再执行 Turnstile、限流、邮箱校验等逻辑。

**前端**

1. 在最后一步「完成注册」前，若后端要求图形验证码，则展示验证码并随注册请求提交。

**说明**：若已启用 Turnstile 且发送验证码已用图形验证码防刷，本步可延后或不做。

---

### 步骤 3：完善 Turnstile 配置与说明

**目标**：正式环境必须开启人机验证，避免机器注册。

**后端**

1. 在 `application.yaml` 或 `sa-base.yaml` 中保留/补充说明：
   - `turnstile.site-key`：前端使用（可给前端通过环境变量注入）。
   - `turnstile.secret-key`：后端校验用，**生产环境必填**；未配置时当前逻辑为跳过校验（仅适合本地开发）。
2. 可选：增加配置项 `turnstile.required=true`，为 true 时未配置 secret-key 则启动报错，避免生产误未配置。

**前端**

1. 确保 `.env` / 构建环境中有 `VITE_TURNSTILE_SITE_KEY`，与 Cloudflare 控制台一致。
2. 注册页已按 `TURNSTILE_SITE_KEY` 是否存在决定是否展示 Turnstile；保持即可。未配置时仍可注册（与后端「未配置则跳过」一致）。

**验收**：生产环境配置 site-key + secret-key 后，未完成人机验证无法注册；本地不配置时可正常注册。

---

### 步骤 4：发送验证码限流（建议与步骤 1 一起做）

**目标**：同一邮箱/同一 IP 在短时间内不能频繁请求验证码，防止轰炸与滥用。

**后端**

1. **按邮箱限流**
   - Key：`support:email:send:rate:{email}`，值：次数或 1，过期时间 60 秒。
   - 在 `EmailVerifyService.sendVerifyCode` 开头：若 key 存在则直接返回“发送过于频繁，请 1 分钟后再试”。
   - 发送成功后 set key，过期 60 秒。

2. **按 IP 限流**
   - Key：`support:email:send:ip:{clientIp}`，值：计数，过期 60 秒。
   - 在 `LoginController.sendVerifyEmail` 中获取 clientIp，传入 service；若该 IP 在 1 分钟内请求超过 N 次（如 5 次），则拒绝。

**前端**：无需改逻辑，后端返回“发送过于频繁”时，前端提示即可。

**验收**：同一邮箱 1 分钟内多次点击「获取验证码」仅第一次成功；同一 IP 分钟内超过 N 次全部被拒。

---

### 步骤 5：谷歌登录（OAuth2）后端接入

**目标**：用户可通过「使用 Google 登录」跳转到 Google 授权，回调后在本系统创建/绑定用户并登录。

**后端（casalyin-iam）**

1. **依赖与配置**
   - 引入 Spring Security OAuth2 Client（或现有安全框架的 OAuth2 支持）。
   - 配置 `spring.security.oauth2.client.provider.google`、`spring.security.oauth2.client.registration.google`（client-id、client-secret、scope 等）。

2. **授权入口**
   - 实现接口：`GET /auth/oauth2/authorize/google`（与前端 `getOAuthAuthorizeUrl('google')` 一致）。
   - 逻辑：重定向到 Google 授权页（state 中可带 redirect_uri、非ce 等）。

3. **回调处理**
   - 实现：`GET /auth/oauth2/callback/google`（或由 Spring OAuth2 默认路径处理）。
   - 用 code 换 token，拿到用户 openid/email 等信息。
   - 若本系统已存在该邮箱用户：直接登录（发 token/session）；若不存在：可选「自动以该邮箱注册」或「要求先绑定已有账号/补充信息再登录」。
   - 自动注册时需确定默认角色（如 `RoleConstant.DEFAULT_REGISTER_ROLE_CODE`）、登录名用邮箱等，与现有 `UserService.register` 逻辑对齐。

4. **用户表扩展（若需要）**
   - 若需区分「仅密码登录」与「OAuth 绑定」，可增加字段如：`oauth_provider`、`oauth_open_id`；或先用邮箱唯一关联，后续再扩展。

**前端（casalyin-server）**

1. 登录页已有「使用 Google 登录」时，使用 `getOAuthAuthorizeUrl('google')` 跳转。
2. 回调后：后端重定向到前端登录成功页（或带 token 的 URL），前端读取 token 或 session 完成登录状态。

**验收**：点击 Google 登录 → 跳转 Google → 授权后回到站点并处于已登录状态；新用户能自动注册或按产品策略引导绑定。

---

### 步骤 6：其他第三方登录（Microsoft、Apple 等）

**目标**：与步骤 5 类似，按需增加 Microsoft、Apple 等 OAuth2 提供商。

**实现要点**

- 后端：新增 `GET /auth/oauth2/authorize/microsoft`、`/auth/oauth2/authorize/apple` 及对应 callback；配置各自 client-id/client-secret。
- 前端：`getOAuthAuthorizeUrl('microsoft')`、`getOAuthAuthorizeUrl('apple')` 已预留，只需后端提供对应路径。
- 用户关联策略与 Google 一致：按邮箱或 openid 关联/自动注册。

---

### 步骤 7：注册成功后的增强（可选）

- **邮件通知**：注册成功后给用户发一封「欢迎注册」邮件。
- **风控/审计**：记录注册 IP、时间、角色等，便于审计与风控。
- **协议勾选**：注册前必须勾选《用户协议》《隐私政策》，表单与后端各增加一项校验。

## 三、建议实施顺序

| 顺序 | 步骤 | 优先级 | 说明 |
|------|------|--------|------|
| 1 | 步骤 1 + 步骤 4 | 高 | 图形验证码 + 发验证码限流，先防刷与轰炸 |
| 2 | 步骤 3 | 高 | 确认 Turnstile 生产配置，避免机器注册 |
| 3 | 步骤 2 | 中 | 注册提交前图形验证码，可按需做 |
| 4 | 步骤 5 | 中 | 谷歌登录，提升体验 |
| 5 | 步骤 6 | 低 | 其他第三方登录按产品需求 |
| 6 | 步骤 7 | 低 | 邮件、审计、协议等增强 |

## 四、相关代码位置速查

- **后端**
  - 注册入口：`LoginController.register` → `UserService.register`
  - 注册表单：`RegisterForm`
  - 发送验证码：`LoginController.sendVerifyEmail` → `EmailVerifyService.sendVerifyCode`
  - 图形验证码：`LoginController.getCaptcha`、`CaptchaService.generateCaptcha` / `checkCaptcha`（sa-base）
  - Turnstile：`TurnstileVerifyService.verify`、`application.yaml` 中 `turnstile.*`
  - 限流：`UserService.register` 内 `REGISTER_RATE`、RedisKeyConst.Support
- **前端**
  - 注册页：`src/pages/Register/index.tsx`
  - 认证 API：`src/api/auth.ts`（`apiGetCaptcha`、`apiSendVerifyEmail`、`apiAuthRegister`、`getOAuthAuthorizeUrl`）

按上述步骤逐个落地即可系统性地把验证码、限流、Turnstile 和谷歌登录等补齐。
